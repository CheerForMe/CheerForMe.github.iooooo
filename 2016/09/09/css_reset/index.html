一列定宽与一列自适应

float + margin
  • 代码结构
float + margin + (fix)
  • 优化float + margin的方法

  • 缺点：IE6不支持

flex
  • 缺点：兼容性；性能可能会有问题

两列定宽与一列自适应

  • 与一列定宽一列自适应基本相同

一列不定宽与一列自适应

float + overflow
  • left的宽度随意
  • 兼容性：IE6不行
table
  • left的宽度随意
  • 如果left的宽度由里面的内容决定，需要删除table-layout:fixed;
  • IE6、7不兼容
flex
  • left宽度随意

多列不定宽与一列自适应

  • 与一列不定宽基本相同

等分

  • 每一列跨度一样、间距一样

float
  • IE6/7中对浮点小数会四舍五入
  • 缺点：当列数变化时，结构与样式的耦合性太强
table
flex

等高

  • 所有元素无论内容多少，高度相同
table
  • 同一列定宽一列自适应
flex
  • 同一列定宽一列自适应
float + overflow
  • 伪等高
  • 兼容性较好。

全屏布局

  • 解决方案：position、flex

position

position兼容
  • IE6不支持
  • 解决方案：全屏自适应-兼容

flex

flex兼容性
  • IE9及以下不支持

宽度/高度百分比

  • 将之前定高的px数值改为百分比即可

宽度/高度由内容决定时

  • position不适合，无法决定中间区域的top、bottom值
  • flex支持
  • grid支持（目前只是w3c草案，绝大多数浏览器不支持）
flex

总结

方案 兼容性 性能 自适应
position 好（IE6可hack） 好 部分自适应
flex 较差 差 可自适应
grid 差 较好 可自适应
  • flex移动端支持较好，移动端性能差，大布局用flex时滑动时会有卡顿

  • 缺点：资源冗余

viewport

页面优化

  • 提升网页响应速度
  • 对搜索引擎、屏幕阅读器友好
  • 代码提高可读性，可维护性

如何优化

  • 减少请求
  • 减少文件大小
  • 页面性能
  • 可读性可维护性

减少请求

  • 图片合并 – sprite
  • css文件合并
    • 多个css文件合并为一个
    • 少量的css用内链方式
    • 避免使用import的方式引入css文件，每个import都是一个同步的请求

减少图片大小

  • 选择合适的图片格式

耗性能属性

  • expression
  • filter
  • border-radius
  • box-shadow
  • gradients

规范与模块化

  • 多人协作开发的角度展开

规范

  • 文件规范
    • 文件分类
      • 通用类：基础样式、第三方库、控件
      • 业务类：专辑、歌手、排行榜等等的专门的样式
    • 文件引入
      • 行内样式：不推荐，但特殊情况下不阻止
      • 外联引入：link标签把通用的样式引入
      • 内联引入：当前页面的相关样式写到style标签里
      • 避免在CSS中使用@import，有渲染问题
    • 文件本身
      • 文件名：命名约束
      • 编码：UTF-8
  • 注释规范
    • 块状注释
      • 统一缩进
      • 放在被注释对象的上面
    • 单行注释
      • 文字两端需空格
      • 在被注释对象之上
    • 行内注释
      • 文字两端需空格
      • 在分号后面
  • 命名规范(CSS)
    • 分类命名
      • 谨防类名相同导致的样式污染
    • 命名格式
      • 大小写建议小写
      • 选择器长度：权衡长度和可读性，既不能太长也不能因为太短导致丧失可读性
    • 语义化命名
      • 以内容语义命名
  • 书写规范
    • 选择器单行还是多行
    • 空格与分号
      • 缩进的空格数
      • 规则里的空格
      • 分号不要省略
    • 属性顺序
      • 根据属性的重要性按顺序书写：显示属性 ==> 自身属性 ==> 文本属性和其他修饰
    • Hack方式
      • 统一各浏览器的Hack方式：IE6的hack前加下划线；IE6/7的Hack前加星号
      • 不要滥用Hack
    • 值格式
      • 颜色值：关键字？16进制？RGB？
      • url()：是否有引号？单引号？双引号？
  • 其他规范（HTML、图片）
    • HTML规范
      • 文档声明：<!doctype html>，首行顶格开始
      • 标签闭合：
      • 属性：双引号？单引号？
      • 层级：缩进、标签正确嵌套
      • 注释
      • 大小写：标签属性都小写
    • 图片规范
      • 文件名：语义、长度
      • 保留源文件
      • 图片合并：尽可能使用sprite；按模块、业务划分

模块化

  • 一系列相关联的结构组成的整体
  • 带有一定语义，而非表现

怎么做？

  • 为模块分类命名
  • 以一个主选择器开头（模块根节点）
  • 使用以主选择器开头的后代选择器（模块子节点）
  • 可扩展：添加一个新的扩展class类名，用扩展+子节点进行新增部分的定义

为什么要模块化

  • 利于多人协同开发
  • 便于扩展和重用
  • 可读性、可维护性
